# **Data Wrangling 案例研究**

## 医疗数据实战演练

预测病人再入院（Patient Readmission）
FIT5196 - Data Wrangling

---

## 🎯 案例背景

**目标：** 建立一个预测模型，判断哪些病人在出院后 **30 天内可能再入院**

**意义：**

- 减少医疗成本
- 提升护理质量
- 优化资源配置

---

<!-- _class: lead -->

# 第一步：🔍 Data Discovery

## 数据发现

---

## 📊 问题：数据源识别

假设你是医院数据分析团队成员，你有哪些可能的数据来源可以使用？（多选）

**A.** 电子病历（EHR）
**B.** 医保索赔数据  
**C.** 病人佩戴的手环数据（可穿戴设备）
**D.** 医生的主观经验描述
**E.** 健康史 & 药物记录

---

## ✅ 答案：全部可用 (A~E)

| 数据源               | 包含内容             | 作用                        |
| -------------------- | -------------------- | --------------------------- |
| **电子病历（EHR）**  | 诊断、用药、检验结果 | 识别既往病史和治疗效果      |
| **医保索赔数据**     | 就诊、费用、报销记录 | 分析经济负担和医疗频率      |
| **可穿戴设备数据**   | 心率、运动、睡眠等   | 分析恢复情况和生活习惯      |
| **医生描述**         | 主观印象、注意事项   | 加入非结构化文本特征（NLP） |
| **健康史与药物记录** | 慢病情况、长期用药   | 评估基础健康风险            |

# 第二步：📥 Data Collection

## 数据收集

---

## 📊 问题：关键因素识别

在收集 EHR 数据时，我们要特别注意哪三项 **关键因素**？
从以下选项中挑出最重要的三个：

**A.** 数据格式（如 CSV、JSON）
**B.** 法律合规与隐私保护（如 HIPAA）
**C.** 数据字段是否统一命名
**D.** 项目预算
**E.** 是否包含关键字段（出院时间、诊断码等）

---

## ✅ 答案：B、C、E

| 关键因素                  | 原因                                          |
| ------------------------- | --------------------------------------------- |
| **B. 法律合规与隐私保护** | 医疗数据高度敏感，必须合法合规处理            |
| **C. 字段命名和一致性**   | 数据来自不同医院/系统，字段可能不统一，需统一 |
| **E. 是否包含关键字段**   | 必须有"再入院"标签才能训练预测模型            |

# 第三步：🛠 Data Pre-processing

## 数据预处理

---

## 📊 问题：识别预处理需求

以下是一份出院记录数据样本：

```csv
patient_id, Discharge_Date, Diagnosis, Readmitted, Age, Gender
P001, 12-03-2024, 心衰, Yes, 67, M
P002, 03/10/2024, Asthma, No, 54, Female
P003, 2024.05.28, Diabetes, YES, 60, F
```

**这个数据有什么需要预处理的地方？**
说出两个你注意到的问题。

---

## ✅ 答案：格式不一致问题

### 🧼 问题 1：列值不一致

- `Gender` 中有 `M` 和 `Female`
- `Readmitted` 有 `Yes` 和 `YES`

**解决方案：** 统一为标准格式

- `gender`: `male` / `female`
- `readmitted`: `yes` / `no`

---

### 📅 问题 2：日期格式混乱

- `12-03-2024` / `03/10/2024` / `2024.05.28`

**解决方案：** 统一为 `YYYY-MM-DD` 格式

# 第四步：🧹 Data Cleaning

## 数据清洗

---

## 📊 问题：区分清洗操作

哪些属于清洗操作，而不是预处理或转换？（多选）

**A.** 删除重复的病人记录
**B.** 把"YES"改成"yes"  
**C.** 把 `Age` 的缺失值填为平均年龄
**D.** 把"心衰"翻译成"Heart Failure"
**E.** 把 `Gender` 列转换成 0/1 数值

---

## ✅ 答案：A、B

| 选项                           | 属于清洗吗？ | 原因                                   |
| ------------------------------ | ------------ | -------------------------------------- |
| ✅ **A. 删除重复记录**         | 是           | 清理重复值，避免偏差                   |
| ✅ **B. 标准化 "YES" ➝ "yes"** | 是           | 统一分类标签，避免误判                 |
| 🔸 **C. 缺失值填补**           | 不完全       | 更偏向"预处理"或"修复"阶段的策略性操作 |
| ❌ **D. 翻译成英文**           | 否           | 属于"数据转换"或"语义统一"             |
| ❌ **E. 类别转数值**           | 否           | 这是"数据转换"常用做法（如编码、映射） |

---

# 第五步：🔄 Data Transformation

## 数据转换

---

## 📊 问题：数值转换识别

我们需要把 `Age` 字段处理为模型更易接受的形式。
以下哪几种处理方式属于"数值转换"？（多选）

**A.** 标准化为 Z-score（均值为 0，标准差为 1）
**B.** 转成年龄段（如 50-59，60-69）
**C.** 正规化到 0~1 之间
**D.** 按年龄生成 "青年""中年""老年"三个标签
**E.** 替换为出生年份

---

## ✅ 答案：A、C

| 选项                     | 是否数值转换？ | 理由                                       |
| ------------------------ | -------------- | ------------------------------------------ |
| ✅ **A. Z-score 标准化** | 是             | 经典的数值变换，适用于线性模型             |
| ✅ **C. 0~1 正规化**     | 是             | 属于 Min-Max scaling，也是数值转换         |
| 🔸 **B. 年龄段划分**     | 有争议         | 更像是"离散化"或"分箱"，用于简化特征       |
| ❌ **D. 青年/中年/老年** | 否             | 属于"语义分类"或"标签化"                   |
| ❌ **E. 替换为出生年份** | 否             | 虽然还是数字，但引入了新含义，是"衍生特征" |

# 第六步：🔧 Data Enrichment

## 数据增强

---

## 📊 问题：特征扩展

如果我们想 enrich "药品使用记录"，我们可以从哪些公开数据或文档中**补充有用信息**？

**例如：** 药品名 ➝ ?

说一个例子即可。

---

## ✅ 答案：药品分类信息

**药品种类（例如：降压药、抗生素、镇痛药）** 是非常实用的 enrich 信息。

### 🧬 药品信息 Enrichment 示例

| 原始字段                     | Enriched 字段                     | 说明                       |
| ---------------------------- | --------------------------------- | -------------------------- |
| `Drug_Name = "Metformin"`    | `Drug_Type = "Antidiabetic"`      | 对分类模型更有帮助         |
| `Drug_Name = "Atorvastatin"` | `Target_Organ = "Cardiovascular"` | 可用于主题分析             |
| `Drug_Name = "Ibuprofen"`    | `OverTheCounter = true`           | 判断病人是否常依赖非处方药 |

**数据来源：** 药典数据库、药品编码系统（如 ATC code）、政府医疗指南

---

### 📦 地理位置信息扩展示例

从 **Postcode** 可扩展出的新特征：

| 衍生字段           | 来源             | 作用                         |
| ------------------ | ---------------- | ---------------------------- |
| **Suburb**         | 地址库           | 细化地理位置，利于地图可视化 |
| **State / Region** | 邮编对照表       | 用于政策/医疗覆盖范围分析    |
| **人口密度**       | 人口普查数据     | 分析城市 vs 郊区患者的差异   |
| **平均收入水平**   | ABS (澳洲统计局) | 社经状态与再入院率关联       |
| **医院距离**       | GIS 地理系统     | 距离远近是否影响复诊率       |

# 第七步：✅ Data Validation

## 数据验证

---

## 📊 问题：验证操作识别

以下哪些操作属于"数据验证"？（多选）

**A.** 检查 `Age` 是否都在 0~120 之间
**B.** 删除空值行
**C.** 确认 `patient_id` 无重复
**D.** 用医生手工标注验证模型输出
**E.** 检查 `Discharge_Date` 不晚于 `Readmit_Date`

---

## ✅ 答案：A、B、C、E

| 选项                       | 属于验证吗？ | 说明                           |
| -------------------------- | ------------ | ------------------------------ |
| ✅ **A. 值范围检查**       | 是           | 避免不合逻辑值影响结果         |
| ✅ **B. 删除空值行**       | 是           | 关键字段不能为空               |
| ✅ **C. 唯一性检查**       | 是           | 主键校验，防止混淆             |
| ✅ **E. 逻辑一致性**       | 是           | 保证时间顺序合理               |
| ❌ **D. 人工验证模型输出** | 否           | 属于"模型评估"或"标注质量检查" |

# 🧠 总结复习卡

## DW 任务全流程

| 步骤                     | 关键词               | 技巧点                   |
| ------------------------ | -------------------- | ------------------------ |
| **1. 发现 Discovery**    | 多源识别             | 判断可访问性、字段质量   |
| **2. 收集 Collection**   | 隐私、格式、字段覆盖 | 合法合规收集重要特征     |
| **3. 预处理 Preprocess** | 格式、子集、抽样     | 统一格式，减小数据量     |
| **4. 清洗 Cleaning**     | 缺失、重复、异常值   | 保障数据可靠             |
| **5. 转换 Transform**    | 标准化、词向量、分箱 | 转换为模型友好结构       |
| **6. 增强 Enrichment**   | 加字段、补语义       | 更丰富地反映现实         |
| **7. 验证 Validation**   | 规则、范围、唯一性   | 检查可用性               |
| **8. 存储 Storing**      | 结构设计、备份、安全 | 保证后续查询和分析可持续 |
